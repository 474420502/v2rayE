# 2026-03-09 TUI First 迁移与 VPN 网络设计

## 1. 目标

本项目当前目标从“Web 管理面板 + 后端 API”切换为“Termshark 风格 TUI + 本机控制平面 + xray-core 数据平面”。

这意味着：

1. `web/` 不再是产品主入口。
2. `backend-go/cmd/tui` 成为默认 UI。
3. 后端 HTTP API 的职责收缩为本机控制面，而不是长期维护一个浏览器前端契约。
4. VPN 设计重心从“页面拼装”转向“核心生命周期、路由模型、TUN 接管、日志观测、排障闭环”。

## 2. 为什么去掉 Web 主路径

当前 Web 路径的问题不是不能用，而是系统边界不够干净：

1. 本机 VPN 工具本质上是终端/桌面控制程序，不需要先经过浏览器交互。
2. `backend + next dev + node scripts` 让开发、运行、排障链条偏长。
3. 浏览器前端容易把很多本应是本机状态机的问题伪装成页面状态问题。
4. TUN、系统代理、路由修复、SSE 日志这类功能，本来就更适合常驻终端控制台表达。

## 3. 目标架构

### 3.1 三层模型

1. UI 层：`backend-go/cmd/tui`
   - 基于 `gowid + tcell`
   - 风格向 termshark 靠拢
   - 负责用户交互、实时观测、排障操作

2. 控制平面：`backend-go`
   - 承担状态存取、配置管理、核心编排、TUN 修复、路由诊断
   - 通过本机 HTTP API 暴露能力
   - 默认只服务本机 TUI，不以浏览器多端访问为设计中心

3. 数据平面：`xray-core`
   - 负责实际代理、路由命中、TUN 流量接管、统计采集
   - 保持单一正式引擎路径，避免多引擎心智负担

### 3.2 关键原则

1. 单一数据平面：只保留 `xray-core` 作为正式 VPN 引擎。
2. 单一主入口：只保留 TUI 作为默认 UI。
3. 后端本地化：HTTP API 面向本机控制，不为浏览器体验妥协核心设计。
4. 观测优先：状态、日志、事件、路由命中、TUN 接管状态必须在主界面可见。
5. 修复闭环：每个高风险网络动作都要有“检查 -> 诊断 -> 修复 -> 再检查”的路径。

## 4. 更合理的 VPN 网络设计

### 4.1 控制平面与数据平面分离

要避免把 TUI 自己的 API 请求错误地当成 VPN 用户流量。

建议规则：

1. TUI 访问后端固定走 `127.0.0.1:18000`，视为控制流量。
2. 本地 HTTP/SOCKS 入口固定由 `xray-core` 提供，视为代理数据流量。
3. TUN 接管只影响目标业务流量，不应干扰本机控制通道的可用性。

### 4.2 路由模型应显式分层

建议统一成以下顺序：

1. 本机保留流量：`localhost`、控制面、必要系统服务
2. 私网与局域网直连流量
3. CN / geosite / geoip 直连或旁路流量
4. 用户自定义路由规则
5. 默认出口策略：`proxy` 或 `direct`

这样做的好处：

1. 避免 TUI/后端被自己代理自己。
2. 避免 TUN 接管后把管理流量也卷进去。
3. 诊断时更容易解释“为什么这个目标走了某个 outbound”。

补充：当前已支持 `routing.localBypassEnabled` 开关（默认 `true`）。

1. `true`：强制前置 `localhost + loopback` 旁路（推荐）。
2. `false`：不注入这两条保底旁路，交由用户规则决定。

仅在特殊调试场景建议关闭，常规运行关闭会增加控制面被代理链路影响的风险。

### 4.3 TUN 设计应强调可恢复性

TUN 是系统最高风险部分，必须按“失败也能撤回”的思路设计：

1. 启动前保存关键路由上下文。
2. 启动后明确暴露：是否接管成功、默认路由设备是什么、策略路由是否就位。
3. 停止或异常退出后，必须优先恢复默认路由和系统代理。
4. TUI 里必须提供一键修复，并展示修复结果而不是只返回成功/失败。

### 4.4 日志与事件模型应服务排障

日志至少要区分：

1. `app`
2. `xray-core`
3. 后续可继续扩展：`tun`、`routing`、`proxy`

事件至少要覆盖：

1. 核心启停与异常退出
2. profile 切换
3. 订阅更新
4. 路由配置变更
5. TUN 修复动作

## 5. termshark 风格集成原则

这里的“使用 termshark UI”建议理解为：

1. 采用与 termshark 一致的终端 UI 技术路线：`gowid + tcell`
2. 学 termshark 的交互模型，而不是强行把抓包功能嫁接进 VPN 控制台
3. 在必要时可借鉴 termshark 的布局、快捷键、面板切换、状态栏和 minibuffer 设计

不建议的路径：

1. 让 VPN 主界面依赖 `tshark` 抓包能力才能运行
2. 为了“像 termshark”而引入与 VPN 主流程无关的大量复杂依赖

## 6. 分阶段迁移建议

### 阶段 1：入口切换

1. 默认开发启动改为 `backend + TUI`
2. `web/` 不再参与默认启动和默认构建
3. README 与脚本说明改为 TUI-first

### 阶段 2：交互收敛

1. 把核心控制、profiles、network、logs 做成稳定主面板
2. 加强快捷键与焦点管理
3. 把关键诊断信息固定在 UI 主路径内

### 阶段 3：网络设计收敛

1. 明确控制流量旁路原则
2. 收敛 routing rule 生成顺序
3. 稳定 TUN 修复与状态回读
4. 用路由测试能力验证默认策略是否可解释

### 阶段 4：历史界面下线

1. 停止维护 `web/` 的功能对齐
2. 把必要的 Node 脚本迁移到 Go 或 shell
3. 最终移除 `web/` 与其构建依赖
